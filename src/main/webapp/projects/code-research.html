Everything here has shaped my thinking about coding in some fashion. There is no suggested order to read any of this, but the most recent ones are first.

# Go Project Layout

My preferred organization for Go projects.

# Complexity Budget

Being able to say "No".

# Splitting Your APIs

I've seen many times when a front-end team demands a general purpose API from a back-end team. This leads to inefficient data population and opaqueness on page load needs. Tailoring a back-end API for the intended front-end and having a generic API sidecar (if necessary) is a great idea.

# Locality of Behavior

Related code spread out leads to missed understanding and unintended side effects.

# Computer Ergonomics Kieran Drew

This is what pushed me over the fence into getting a nicer chair and standing desk. Code suffers if the author suffers.

# Architectural Decision Records Bruno Scheufler

In many projects the context for decisions have disappeared long before the effects of the decision. I think ADR's are a great documentation format to adopt even in teams of one to help record the context of choices. More information can be found through the ADR Github Organization.

# Choose Boring Technology Dan McKinsley

Developers often push for radically changing working architecture by adding new tools. More of their writing can be found under The .Club Club. Almost all say how I feel better than I ever could.

# The Grug Brained Developer

The first few paragraphs were admittedly hard to get into the groove of reading, but it picks up quickly. The section on testing has made me think more about integration tests over my preferred unit tests. The writer of this also created HTMX which I am interested in.

# Local First Development Ink & Switch

A relatively lengthy read on the ideals of offline and collaboration capable software. The writing structure walks through seven benefits of this approach with occasional deep dives into technical aspects. This also introduced me to Conflict-free Replicated Data Types (CRDTs).

# The Joel Test Joel Spolsky

A nice easy-to-understand checklist of things that teams should be doing. There are a few parts that I'm not sure about such as daily builds instead of continuous integration. Some pieces of it are more geared to project managers such as scheduling, but all of it is adaptable.

# Code: The Hidden Language of Computer Hardware and Software Charles Petzold

A connection between the "real world" and the programming used to facilitate digital communication.
